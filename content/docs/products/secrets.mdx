title: Secrets
description: Store, rotate, version, and consume sensitive project values.

Secrets stores encrypted values for a project. Container services, game presets, environment variables, Git credentials, and database credentials can reference a secret without placing its plaintext value in the workload configuration.

## Create a secret

Open **Secrets** and select **Create Secret**. Supply:

- A name, normalized to uppercase letters, numbers, and underscores
- The initial value
- An optional description and comma-separated tags
- An environment label, which defaults to `default`
- An optional rotation interval in days

Values are not returned in list responses.

## Reveal and update

Open a secret to reveal its current value. Reveal is an audited operation and requires secret-value read permission. Hide the value after use and avoid copying it into ordinary environment-variable fields.

Saving a new value creates an immutable version. A change note is optional. **Rotate** can use the value you entered or generate a replacement when the new-value field is empty; consumers pinned to the latest value are then refreshed by the backend.

## Version history and rollback

Version history identifies create, update, rotate, and rollback events. Rolling back does not overwrite history: it saves the selected old value as a new current version.

## Delete

Deleting a secret permanently removes every version. Remove service and database references first; the confirmation dialog requires the secret name.

<Callout type="warn" title="Rotation schedules are metadata in the frontend">
  You can set a rotation interval and see the next rotation timestamp, but the frontend does not expose a scheduler status page or rotation-job history. Verify automated-rotation behavior in the deployment where it is enabled.
</Callout>
